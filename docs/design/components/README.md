# コンポーネント設計

## 1. 概要
本ディレクトリでは、Taskuestにおけるコンポーネント設計を管理する。  
UIの一貫性および保守性を高めるため、共通UIコンポーネント・共通レイアウト・業務コンポーネントを分離して定義する。

---

## 2. ディレクトリ構成

```txt
components/
  ui/
    button.md
    input.md
    textarea.md
    modal.md
    toast.md
  layout/
    header.md
    sidebar.md
    user-menu.md
    authenticated-layout.md
  feature/
    （業務コンポーネントを配置）
```

- `ui/`  
  汎用的に再利用されるUIコンポーネントを配置する  
  （Button, Input などの小さな部品）

- `layout/`  
  画面全体の構造や配置を担う共通レイアウトを配置する  
  （Header, Sidebar などの骨組み）

- `feature/`  
  特定の業務や画面に依存するコンポーネントを配置する  

---

## 3. 設計方針

### 3.1 責務分離
- UIコンポーネントは表示と基本操作のみを担当する
- レイアウトコンポーネントは画面構造と配置を担当する
- 業務ロジックはfeature側のコンポーネントで扱う

---

### 3.2 共通UIコンポーネント
以下の条件を満たす場合、共通UIとして定義する

- 複数画面で再利用される
- 業務知識に依存しない
- 見た目および基本操作を統一したい

例：
- Button
- Input
- Textarea
- Modal
- Toast

---

### 3.3 共通レイアウト
以下の条件を満たす場合、共通レイアウトとして定義する

- 複数画面で使用される画面構造を持つ
- UIコンポーネントを組み合わせて構成される
- ページ全体の配置や導線に関わる

例：
- Header
- Sidebar
- UserMenu
- AuthenticatedLayout

---

### 3.4 業務コンポーネント
以下の条件を満たす場合、featureコンポーネントとして定義する

- 特定の画面や機能に依存する
- 業務状態（例：タスク完了状態）を扱う
- UIコンポーネントやレイアウトを組み合わせて構成される

例：
- TaskCompleteButton
- TaskFilterBar
- TaskCard

---

### 3.5 スタイル管理
- 基本スタイルはコンポーネント側で定義する
- 個別調整が必要な場合のみ className を使用する
- 同一の調整が複数箇所で発生する場合は共通化を検討する

---

### 3.6 variant の扱い
- variant はUI上の役割（主操作・補助操作・危険操作）を表す
- 業務種別（例：完了、アーカイブ）は含めない
- variant は必要なコンポーネントにのみ限定して導入する

---

### 3.7 共通化の判断基準
- 2箇所以上で利用される場合に共通化を検討する
- 将来の利用を想定した過剰な抽象化は行わない
- 必要になったタイミングで共通化する

## 5. 運用ルール

- コンポーネントは原則として「1ファイル1コンポーネント」で定義する
- 新規コンポーネント追加時は本READMEにリンクを追加する
- 共通化の判断は「複数箇所で利用されるか」を基準とする
- layout と ui を混在させない
