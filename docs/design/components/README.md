# コンポーネント設計

## 1. 概要
本ディレクトリでは、Taskuestにおけるコンポーネント設計を管理する。  
UIの一貫性および保守性を高めるため、共通コンポーネントと業務コンポーネントを分離して定義する。

---

## 2. ディレクトリ構成

```txt
components/
  ui/
    button.md
  feature/
    （業務コンポーネントを配置）
```

- `ui/`  
  汎用的に再利用されるUIコンポーネントを配置する

- `feature/`  
  特定の業務や画面に依存するコンポーネントを配置する

---

## 3. 設計方針

### 3.1 責務分離
- UIコンポーネントは表示と基本操作のみを担当する
- 業務ロジックはfeature側のコンポーネントで扱う

---

### 3.2 共通UIコンポーネント
以下の条件を満たす場合、共通UIとして定義する

- 複数画面で再利用される
- 業務知識に依存しない
- 見た目および基本操作を統一したい

例：
- Button
- Input
- Select

---

### 3.3 業務コンポーネント
以下の条件を満たす場合、featureコンポーネントとして定義する

- 特定の画面や機能に依存する
- 業務状態（例：タスク完了状態）を扱う
- UIコンポーネントを組み合わせて構成される

例：
- TaskCompleteButton
- TaskFilterBar

---

### 3.4 スタイル管理
- 基本スタイルはコンポーネント側で定義する
- 個別調整が必要な場合のみ className を使用する
- 同一の調整が複数箇所で発生する場合は共通化を検討する

---

### 3.5 variant の扱い
- variant はUI上の役割（主操作・補助操作・危険操作）を表す
- 業務種別（例：完了、アーカイブ）は含めない

---

### 3.6 iconOnly の扱い
- アイコンのみ表示する場合は `iconOnly` プロパティで制御する
- 表示形式として扱い、variant や size とは独立させる

## 5. 運用ルール

- コンポーネントは原則として「1ファイル1コンポーネント」で定義する
- 新規コンポーネント追加時は本READMEにリンクを追加する
- 共通化の判断は「複数箇所で利用されるか」を基準とする
