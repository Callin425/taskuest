# ADR-0006: Issue / Pull Request 運用方針（大Issue / 小Issueの分離）

## ステータス

採用

## 日付

2026-04-23

---

## 背景

Taskuest の開発において、以下の課題が発生した。

- Issue の粒度と Pull Request の粒度が一致しない
- 1画面単位での開発（feature/task-list など）を採用しているため、PRは画面単位でまとめたい
- 一方で、コンポーネント単位・作業単位での進捗も把握したい
- Issue を細かく分割した場合、PRと1対1で対応させるとPR数が増加し、1人開発において運用コストが高くなる

このため、
**「成果物単位の管理」と「作業単位の進捗管理」を両立する運用方法**の検討が必要となった。

---

## 決定

以下の運用を採用する。

- Issue を「大Issue（親）」と「小Issue（子）」に分割する
- 大Issueは画面単位または機能単位の成果物として扱う
- 小Issueはその成果物を構成する個別作業として扱う

クローズ方法は以下とする。

- 大Issue：
  - 対応するPRが dev ブランチにマージされた時点でクローズする（`closes` を使用）
- 小Issue：
  - 作業完了時に手動でクローズする
  - PRとは `refs` により関連付ける

---

## 検討した案

### 案1: IssueとPRを1対1で対応させる

- 各Issueごとにfeatureブランチを作成し、PRを作成する

例：
- Buttonコンポーネント → PR
- TaskCardコンポーネント → PR

---

### 案2: 画面単位で1つのIssueと1つのPRで管理する

- Issueを細分化せず、画面単位のIssueのみ作成
- PRも画面単位で1つ

---

### 案3: 大Issue / 小Issueを分け、クローズ方法を分離する（採用）

- 親子関係でIssueを管理
- PRは画面単位でまとめる
- 小Issueは手動クローズ

---

## 理由

案3を採用した理由は以下の通り。

- PRを画面単位でまとめることで、レビュー・履歴の可読性が高くなる
- 小Issueにより、作業単位での進捗を可視化できる
- IssueとPRを完全に1対1対応させる必要がなく、1人開発における運用コストを抑えられる
- 成果物単位（大Issue）と作業単位（小Issue）の責務を明確に分離できる

---

## 影響

### メリット

- 進捗が細かく可視化される
- PRが大きな成果単位として整理される
- 1人開発でも過剰なPR分割を避けられる
- 実務に近いIssue管理構造（親子関係）を再現できる

---

### デメリット / 制約

- 小Issueのクローズが手動となるため、運用ルールの徹底が必要
- IssueとPRの対応が完全に一致しないため、初見では理解コストがある
- チーム開発時には運用ルールの共有が必須となる

---

## 備考（任意）

- v2以降、チーム開発を想定する場合は、IssueとPRの粒度を再検討する可能性がある
- 小Issueの粒度が過度に細かくならないよう、適宜見直す
