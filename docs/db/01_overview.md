# DB Overview

## 概要

Taskuest v1 のデータベースは、以下の 5 テーブルで構成する。

- `users`
- `user_progress`
- `projects`
- `project_members`
- `tasks`

この構成により、以下を実現する。

- 個人タスク管理
- プロジェクト単位のタスク管理
- チームメンバー管理
- ロールによるアクセス制御の基盤
- ユーザーの経験値・レベル管理
- アーカイブと論理削除の使い分け

---

## 設計方針

### 1. 認証と業務データを分離する
認証は Supabase Auth に任せ、アプリ側のプロフィール情報は `users` テーブルで管理する。

- 認証情報: Supabase Auth
- アプリ側プロフィール: `users`

これにより、パスワードなどの機密情報をアプリ側テーブルで持たずに済む。

### 2. 個人タスクとプロジェクトタスクを同じテーブルで扱う
`tasks` テーブルで、個人タスクとプロジェクトタスクの両方を管理する。

- 個人タスク: `project_id IS NULL`
- プロジェクトタスク: `project_id IS NOT NULL`

これにより、タスク管理の中心を 1 テーブルに集約できる。

### 3. プロジェクト所属は中間テーブルで管理する
ユーザーとプロジェクトの関係は `project_members` テーブルで管理する。

これにより、以下を表現できる。

- 1 人のユーザーが複数プロジェクトに所属する
- 1 つのプロジェクトに複数ユーザーが所属する
- プロジェクトごとのロール管理を行う

### 4. 成長要素は専用テーブルで分離する
ユーザーの経験値やレベルは `user_progress` テーブルで管理する。

これにより、アカウント情報とゲーム的成長要素を分離できる。

### 5. 削除と保管を分ける
`tasks` では、削除とアーカイブを別の概念として扱う。

- `deleted_at`: 論理削除
- `archived_at`: 保管

これにより、「不要になったもの」と「記録として残したいもの」を区別できる。

### 6. 詳細な監査ログテーブルは v1 では採用しない
v1 では専用ログテーブルは作成せず、最小限の追跡情報を各テーブルに持たせる。

例:
- `created_at`
- `updated_at`
- `deleted_at`
- `created_by_user_id`
- `updated_by_user_id`
- `deleted_by_user_id`

これにより、肥大化を防ぎつつ、最低限の追跡性を確保する。

## テーブル一覧

| テーブル名 | 役割 |
|---|---|
| `users` | アプリ利用者のプロフィール情報を管理する |
| `user_progress` | ユーザーの経験値・レベル・完了実績を管理する |
| `projects` | プロジェクト本体を管理する |
| `project_members` | プロジェクト所属とロールを管理する |
| `tasks` | 個人タスク・プロジェクトタスクを管理する |

---

## テーブル関係の概要

- `users` : `user_progress` = 1 : 1
- `users` : `projects` = 1 : N
- `users` : `project_members` = 1 : N
- `projects` : `project_members` = 1 : N
- `projects` : `tasks` = 1 : N
- `users` : `tasks` = 1 : N

---

## ロール設計

`project_members.role` により、プロジェクトごとの権限を表現する。

想定ロールは以下の通り。

- `owner`
- `admin`
- `member`

### ロールの想定役割
- `owner`: プロジェクト所有者
- `admin`: メンバー管理や主要操作が可能
- `member`: 通常参加メンバー

詳細な操作制御は RLS およびアプリ側ロジックで行う。

---

## RLS 方針

Taskuest v1 では、Supabase の Row Level Security を利用して、ユーザーごとのアクセス制御を行う。

基本方針は以下の通り。

### `users`
- 自分の情報だけ参照・更新可能

### `user_progress`
- 自分の成長情報だけ参照・更新可能

### `projects`
- 自分が owner のプロジェクト、または所属しているプロジェクトのみ参照可能

### `project_members`
- 自分が所属するプロジェクトのメンバー情報のみ参照可能
- 追加・更新・削除は `owner` / `admin` を中心に制御する

### `tasks`
- 個人タスクは所有者本人のみ参照可能
- プロジェクトタスクは所属メンバーのみ参照可能

---

## 補足

### プロジェクト作成時の注意
`projects` 作成時は、作成者を `project_members` にも `owner` として登録する前提とする。

### v1 で採用しないもの
以下は v1 では採用しない。

- 親タスク / 子タスク
- タスク同士の依存関係
- プロジェクト用進捗テーブル
- 専用ログイン履歴テーブル
- 詳細監査ログテーブル

これらは将来拡張候補とする。

---

## この設計の位置づけ

このデータベース設計は、Taskuest の v1 向け構成であり、以下の両立を目的とする。

- ポートフォリオとして見せられる実装
- 将来も育てていけるプロダクトの土台

機能を増やしすぎず、しかし拡張しやすい構成を目指している。
